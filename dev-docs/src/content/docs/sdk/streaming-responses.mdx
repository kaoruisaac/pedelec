`session.onChat()` receives incremental assistant text produced during an active user turn.

## Signature

```ts
const unsubscribe = session.onChat((text, ctx) => {
  // text is one delta
});
```

```ts
type ChatEventContext = PedelecEventContext & {
  type: "chat_delta";
  turnId: string;
  turnStartedAt: number;
  eventReceivedAt: number;
};
```

A delta can be a word, punctuation, several sentences, or any other chunk. Chunk boundaries do not carry semantic meaning.

## Accumulate one response

```ts
let assistantText = "";

const offChat = session.onChat((delta) => {
  assistantText += delta;
  renderAssistantMessage(assistantText);
});

await session.sendText("Explain this screen.");
offChat();
```

Do not create a new message bubble for every delta.

```ts
// Incorrect: produces many tiny messages.
session.onChat((delta) => {
  messages.push({ role: "assistant", text: delta });
});
```

Instead, create one assistant message per turn and append to it.

## Group by turn

```ts
type AssistantMessage = {
  id: string;
  sessionId: string;
  turnId: string;
  text: string;
};

const messages = new Map<string, AssistantMessage>();

session.onChat((delta, ctx) => {
  const key = `${ctx.sessionId}:${ctx.turnId}`;
  const current = messages.get(key) ?? {
    id: key,
    sessionId: ctx.sessionId,
    turnId: ctx.turnId,
    text: "",
  };

  current.text += delta;
  messages.set(key, current);
  renderMessages([...messages.values()]);
});
```

`turnId` is generated by the SDK for one accepted `sendText()` or `prepare()` lifecycle. Chat deltas only surface for user turns. Treat the ID as opaque:

- do not parse timestamps from it;
- do not depend on its prefix;
- do not use it as a provider resume ID; and
- do not expect it to survive a page reload.

## Multiple sessions

Every callback includes `ctx.sessionId`, even though the handler is attached to one session. It is still useful for generic stores and logging.

```ts
session.onChat((delta, ctx) => {
  transcriptStore.append(ctx.sessionId, ctx.turnId, delta);
});
```

When a UI switches active sessions, continue recording deltas for non-visible sessions rather than accidentally appending them to the currently selected transcript.

## Timing fields

- `sessionCreatedAt`: when this browser-side session object was constructed.
- `turnStartedAt`: when the SDK accepted the local turn.
- `eventReceivedAt`: when the SDK received the Core event from the extension.
- `eventEmittedAt`: when the SDK created and delivered the callback context.

These timestamps are useful for diagnostics and UI metrics. They are browser clock values, not a distributed tracing guarantee.

## Rendering performance

Very frequent deltas can cause unnecessary framework renders. Keep correctness first, then batch rendering if needed:

```ts
let buffer = "";
let scheduled = false;

session.onChat((delta) => {
  buffer += delta;
  if (scheduled) return;

  scheduled = true;
  requestAnimationFrame(() => {
    scheduled = false;
    renderAssistantMessage(buffer);
  });
});
```

Do not drop the final text. `sendText()` completion can be used to force one last render if your batching strategy has a separate buffer.

## Unsubscribe and component cleanup

Every registration returns a cleanup function:

```ts
const offChat = session.onChat(handleDelta);

onComponentCleanup(() => {
  offChat();
});
```

Unsubscribing prevents future callback delivery to that handler. It does not stop the provider turn or end the session.

Repeatedly mounting a component without cleanup causes duplicate text appends and memory growth.

## Error and completion boundaries

There is no dedicated “message complete” callback. Use the `sendText()` promise or status returning to `idle` as the turn boundary.

```ts
let text = "";
const off = session.onChat((delta) => (text += delta));

try {
  await session.sendText(prompt);
  saveCompletedAssistantMessage(text);
} catch (error) {
  saveInterruptedAssistantMessage(text, error);
} finally {
  off();
}
```

A failed turn may still have emitted partial text. Decide whether to retain, mark, or discard that partial response.

## HTML and Markdown safety

Deltas are plain text from the provider. If you render accumulated output as Markdown or HTML, apply the same sanitization and content-security rules used for any untrusted model output. Never insert raw output into `innerHTML` without a trusted sanitizer.
